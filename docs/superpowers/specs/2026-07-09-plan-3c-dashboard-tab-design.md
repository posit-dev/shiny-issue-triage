# Plan 3c: Dashboard tab for the review app

**Date:** 2026-07-09
**Issue:** #33 (parent #9)
**Status:** Approved for implementation

## 1. What this is

A "Dashboard" tab added to the existing review-queue Shiny app
(`src/triage_verse/review_app/app.py`) that surfaces the program's health
metrics in one place:

1. **Backlog burndown per repo** — open-issue count sampled at the start of
   each ISO week, for all repos combined or one repo.
2. **Opened vs closed per week** — weekly issue inflow and outflow.
3. **Close-reason mix** — counts of `state_reason` for closed issues.
4. **Triage coverage** — per repo, how many open issues have a
   classification row (i.e. the pipeline has looked at them).
5. **Queue depth and throughput** — how many proposals are currently
   undecided, and how many decisions were recorded per week.
6. **Per-category precision** — of decided proposals, the approval rate per
   action type (`approved / (approved + rejected)`); skips are shown but
   excluded from the rate.
7. **Per-stage spend** — total USD and token counts grouped by pipeline
   stage from the `spend` table.

Metrics 1–3 already exist as pure functions in `src/triage_verse/analytics.py`
and are reused unchanged. Metrics 4–7 are new, small computations over data
that already exists (the `classifications` and `spend` tables in the mirror,
and the proposals/decisions JSONL trees).

**Out of scope:** the slide-over drawer (separate Plan 3 slice), any React or
`shinyreact` frontend, new CLI subcommands, changes to the analytics export
format, auto-refresh/polling (a browser reload re-reads everything), and any
new pipeline instrumentation.

**Definition of done**

1. `shiny run src/triage_verse/review_app/app.py` against a real `.data/`
   directory shows two tabs: the existing Queue (unchanged behavior) and a
   Dashboard rendering all seven metrics.
2. A repo selector on the Dashboard switches burndown / flux / close-reason
   between all-repos and a single repo.
3. All new computation and chart-generation code lives in a pure module with
   full offline unit coverage — no network, no model calls, no Shiny test
   harness.

## 2. Design decisions and alternatives considered

**Charts as server-generated inline SVG.** The app is plain server-rendered
Shiny with no plotting dependency. Three options were considered:

- *matplotlib via `render.plot`* — the standard Shiny approach, but adds a
  heavy dependency to a deliberately lean project for four simple charts.
- *HTML/CSS bars only* — dependency-free but poor for the weekly time series
  (hundreds of points over multi-year history).
- *Inline SVG strings generated by pure Python helpers* — dependency-free,
  handles line and bar charts fine at this scale, and the helpers are unit
  testable as plain string-returning functions. **Chosen.**

**New pure module `dashboard.py` rather than growing `analytics.py`.**
`analytics.py` is scoped to mirror-only burndown/flux series that feed the
JSON export; the new metrics read triage state (classifications, spend,
proposals/decisions JSONL) and exist only for the UI. Keeping them separate
preserves that boundary and follows the repo's 1:1 module/test pattern.

**Skips excluded from precision.** A skip is "not judged", not "wrong";
counting it against precision would penalize deferring hard calls.

## 3. Components

- **`src/triage_verse/dashboard.py`** (new, pure, no Shiny import)
  - `triage_coverage(con) -> list[dict]` — per repo:
    `{repo, open, classified, pct}` where `classified` counts open issues
    with a row in `classifications`. Ends with a `TOTAL` row.
  - `weekly_throughput(decisions_dir) -> list[dict]` — decisions per ISO
    week from `decided_at`: `{week, decided}`.
  - `category_precision(decisions_dir) -> list[dict]` — per action:
    `{action, approved, rejected, skipped, precision}`; `precision` is
    `None` when nothing has been approved or rejected yet.
  - `stage_spend(con) -> list[dict]` — per stage:
    `{stage, calls, input_tokens, cached_tokens, output_tokens, usd}`,
    ordered by USD descending.
  - `svg_line_chart(series, ...) -> str` — a minimal SVG string builder for
    one or two line series, used for burndown and throughput (one series)
    and opened-vs-closed (two series; with multi-year weekly history —
    hundreds of points — lines stay readable where grouped bars would not).
    Empty input returns a short "no data" placeholder string. Colors are the
    validated categorical slots blue `#2a78d6` and aqua `#1baf7a`; because
    aqua sits below 3:1 contrast on a light surface, every chart ships with
    a visible legend and an adjacent data table.
  - Decisions JSONL parsing reuses `review_queue._iter_jsonl_records`
    (promoted to a public name, `iter_jsonl_records`, since two modules now
    need it).

- **`src/triage_verse/review_app/app.py`** (modified)
  - Top level becomes `ui.page_navbar` with two panels: **Queue** (the
    existing UI and server logic, unchanged) and **Dashboard**.
  - Dashboard panel: a repo `input_select` ("All repos" + each repo present
    in the mirror), a row of summary value boxes (queue depth, open issues,
    coverage %, total spend), then sections for burndown, opened-vs-closed,
    close reasons, coverage table, throughput, precision table, and spend
    table.
  - The server computes everything on demand from `_con` and the JSONL
    directories already configured via `TRIAGE_VERSE_*` env vars. Queue
    depth reuses the queue's existing reactive value.

- **`tests/triage_verse/test_dashboard.py`** (new) — seeds a temp mirror via
  `db.connect`/`db.upsert_*`/`db.insert_spend` and temp JSONL trees (same
  fixtures style as `test_review_queue.py`), covering: coverage counts and
  percentages, throughput bucketing by ISO week, precision math including
  the zero-judged case, spend aggregation and ordering, SVG helpers
  producing well-formed output and the empty-data placeholder.

## 4. Data flow

```
mirror.sqlite ──(analytics.py: burndown, flux, close reasons)──┐
mirror.sqlite ──(dashboard.py: coverage, spend)────────────────┤
proposals/ + decisions/ JSONL ──(review_queue + dashboard.py: ─┤──▶ app.py Dashboard tab
                                 depth, throughput, precision)─┘      (tables + inline SVG)
```

Everything is recomputed when the Dashboard outputs render; there is no
cache layer. At current scale (~4,200 issues, low-thousands of JSONL
records) each metric is milliseconds of work.

## 5. Error handling

- Empty mirror / no proposals / no decisions / no spend: every function
  returns an empty list or zero-valued rows, and the UI renders the
  "no data" placeholder rather than erroring.
- Malformed JSONL lines are skipped with a logged warning (existing
  `iter_jsonl_records` behavior).
- Decision records missing `decided_at` or `action` are skipped by the
  metric functions rather than crashing.
